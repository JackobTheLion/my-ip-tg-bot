# Repository Guidelines

## Структура проекта и организация модулей

Это небольшой Telegram-бот на Spring Boot, собираемый через Maven. Код приложения находится в `src/main/java/ru/telegrambot/myiptgbot`.

- `MyIptgBotApplication.java` запускает Spring Boot приложение.
- `config/` содержит настройку старта бота, например `BotInitializer`.
- `controller/` публикует HTTP-эндпоинты, сейчас это `/ip` и интеграция уведомлений Synology.
- `service/` содержит поведение Telegram-бота, получение IP и логику уведомлений.
- `src/main/resources/application.properties` связывает обязательные переменные окружения.
- `Dockerfile` упаковывает собранный JAR в образ Amazon Corretto 17 Alpine.

Директории `src/test` пока нет; добавляйте тесты в `src/test/java/ru/telegrambot/myiptgbot`, повторяя структуру основных пакетов.

## Команды сборки, тестирования и разработки

Используйте Maven Wrapper, чтобы участникам не требовалась локальная установка Maven:

- `./mvnw clean package` собирает JAR приложения в `target/`.
- `./mvnw test` запускает тесты.
- `./mvnw spring-boot:run` запускает приложение локально.
- `docker build -t my-ip-tg-bot .` собирает контейнерный образ после упаковки приложения.

Для локального запуска нужны `BOT_NAME`, `BOT_TOKEN` и `BOT_ADMIN_CHAT`. Пример:

```sh
BOT_NAME=my_bot BOT_TOKEN=123:abc BOT_ADMIN_CHAT=123456 ./mvnw spring-boot:run
```

## Стиль кода и соглашения об именовании

Используйте Java 17 и стандартные соглашения Spring Boot. Оставляйте пакеты внутри `ru.telegrambot.myiptgbot`. Применяйте отступы в 4 пробела, constructor injection для обязательных зависимостей и понятные имена классов с ролью в конце, например `IpController`, `IpClient` или `NotificationService`.

Предпочитайте уже используемые Lombok-аннотации, такие как `@Slf4j` и `@AllArgsConstructor`, когда они убирают шаблонный код и не скрывают поведение. Имена конфигурационных свойств должны соответствовать `application.properties`.

## Правила тестирования

При добавлении тестов используйте JUnit 5 через тестовую поддержку Spring Boot. Называйте unit-тесты по шаблону `*Test`, а интеграционные тесты - `*IT`. В первую очередь покрывайте `IpClient`, обработку Telegram-команд, поведение планировщика и ответы контроллеров.

Перед открытием pull request запускайте `./mvnw test`. Если изменение зависит от Telegram или внешних IP-сервисов, мокайте эти зависимости вместо обращения к live-сервисам в тестах.

## Правила коммитов и pull request

В текущей истории используются короткие и прямые заголовки коммитов, например `Basic functions added` и `CRON ip check added`. Делайте заголовки лаконичными и ориентированными на действие; сначала указывайте фичу или исправление.

Pull request должен содержать краткое описание, результаты проверок вроде `./mvnw test`, связанные issue при наличии и заметки о новых переменных окружения или изменениях деплоя. Скриншоты или примеры HTTP-ответов добавляйте только при изменении поведения эндпоинтов.

## Безопасность и конфигурация

Никогда не коммитьте реальные токены бота, chat ID или сгенерированные секреты. Храните чувствительные значения в переменных окружения и документируйте новые обязательные переменные в `README.md` и в этом руководстве.
